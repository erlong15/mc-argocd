# ArgoCD: GitOps с нуля

---

## 1. Зачем вообще менять подход к деплою

- ручные `kubectl apply` плохо масштабируются
- состояние кластера начинает расходиться с тем, что хранится в Git
- после инцидента сложно понять, кто и что поменял
- rollback зависит от памяти инженера, а не от процесса

**Тезис:** GitOps делает Git источником истины, а кластер — исполняемой копией желаемого состояния.

---

## 2. Что такое GitOps

- инфраструктура и приложения описаны декларативно
- изменения проходят через commit / merge request
- контроллер сам приводит систему к desired state
- drift считается отклонением, а не нормой

---

## 3. Где в этой модели ArgoCD

- ArgoCD читает Git
- рендерит manifests / Helm / Kustomize
- сравнивает desired state и live state
- синхронизирует кластер

**Ключевая идея:** не CI пушит в кластер, а кластер pull-моделью забирает изменения сам.

---

## 4. Push vs Pull

### Push-модель
- GitLab CI / Jenkins получают доступ в кластер
- деплой происходит из pipeline
- контроль разбросан между CI и cluster state

### Pull-модель
- доступ в кластер нужен ArgoCD, а не каждому pipeline
- история изменений централизована в Git
- проще контролировать drift и rollback

---

## 5. Базовая архитектура ArgoCD

- **argocd-server** — UI/API/SSO entrypoint
- **repo-server** — рендер Git/Helm/Kustomize
- **application-controller** — reconciliation loop
- **redis** — кэш и внутренняя служебная роль

Цикл работы:
1. прочитать репозиторий
2. отрендерить манифесты
3. сравнить с live state
4. синхронизировать различия

---

## 6. Ключевые сущности

### Application
Одна логическая единица деплоя.

### AppProject
Границы: какие repo, namespace и cluster разрешены.

### ApplicationSet
Генератор набора Applications по шаблону.

---

## 7. Когда Application уже недостаточно

Один Application подходит, когда:
- одно приложение
- одна среда
- ручное создание допустимо

Нужен ApplicationSet, когда:
- несколько окружений
- много однотипных приложений
- нужен массовый rollout по шаблону

---

## 8. Установка ArgoCD

Варианты:
- raw manifests
- Helm chart
- bootstrap через отдельный GitOps repo

Для мастер-класса:
- быстро ставим manifests
- дальше показываем, как bootstrap-ить через Git

---

## 9. Первая демонстрация

Покажем:
- установку ArgoCD
- логин в UI
- создание приложения
- ручной sync
- auto-sync
- self-heal после ручной поломки Deployment

---

## 10. Структура репозитория

```text
repo/
  charts/nginx-demo/
  environments/dev/
  environments/prod/
  apps/
  bootstrap/
  secrets/
```

Смысл:
- chart содержит шаблоны
- env-файлы содержат настройки окружений
- apps и bootstrap описывают ArgoCD-объекты
- secrets показывают интеграцию с SOPS

---

## 11. Почему Helm, а не просто YAML

- меньше дублирования
- параметры вынесены в values
- проще поддерживать несколько env
- удобно показывать практику chart + values + GitOps

Но:
- chart легко усложнить
- плохой chart превращается в шаблонизатор хаоса

---

## 12. Environments

### dev
- 1 replica
- маленькие ресурсы
- отдельный hostname

### prod
- 2+ replicas
- жестче requests/limits
- другая политика sync

---

## 13. Демонстрация drift

Сценарий:
- приложение в статусе Synced
- руками удалить Deployment из кластера
- ArgoCD покажет OutOfSync
- при self-heal состояние восстановится

Это один из самых наглядных моментов мастер-класса.

---

## 14. Sync policy

- **manual** — безопасно для обучения и контроля
- **automated** — для GitOps-потока
- **prune** — удалять объекты, которых больше нет в Git
- **selfHeal** — исправлять drift без участия инженера

---

## 15. Rollback в GitOps

Не нужно «откатывать кластер руками».

Подход:
- делаем `git revert`
- ArgoCD видит новый desired state
- возвращает систему к прошлой версии

---

## 16. AppProject как граница безопасности

Что ограничиваем:
- список разрешённых Git repo
- список разрешённых cluster destinations
- список namespace
- whitelist / blacklist kinds

**Тезис:** без Project multi-tenant GitOps быстро становится опасным.

---

## 17. ApplicationSet

Показываем шаблон:
- один список env
- на выходе две Application: dev и prod
- у каждой свой values-файл

Это удобная точка входа в масштабирование ArgoCD.

---

## 18. Секреты и почему обычный Git не подходит

Проблема:
- plaintext secret в Git — почти всегда плохая идея

Варианты:
- External Secrets Operator
- Vault
- Sealed Secrets
- SOPS

Для мастер-класса берём **SOPS** как понятный и компактный пример.

---

## 19. Что делает SOPS

- шифрует только чувствительные поля
- файл остаётся YAML
- можно хранить в Git
- расшифровка завязана на age / KMS / PGP

Важно:
- ArgoCD сам по себе SOPS «магически» не понимает
- обычно нужен plugin / CMP / helm-secrets / KSOPS

---

## 20. Что покажем по SOPS

- структуру `.sops.yaml`
- пример age-ключа
- пример зашифрованного секрета
- место интеграции с ArgoCD

На мастер-классе лучше честно сказать:
- для продакшена нужно заранее выбрать supported integration pattern
- хранить приватные age-ключи внутри Git нельзя

---

## 21. Типовые ошибки внедрения

- смешивание app manifests и cluster bootstrap без структуры
- отсутствие AppProject
- auto-sync без понимания prune/selfHeal
- секреты в открытом виде
- ручные изменения в кластере «на всякий случай»

---

## 22. Когда ArgoCD действительно окупается

- несколько сервисов и окружений
- команда уже работает через Git
- нужен audit trail изменений
- есть требования к воспроизводимости и rollback

---

## 23. Когда он будет лишним

- нет Kubernetes
- один стенд, один сервис и ручное управление допустимо
- команда пока не готова к GitOps-дисциплине

---

## 24. Итог мастер-класса

После занятия участник должен:
- понимать модель GitOps
- уметь установить ArgoCD
- создать Application и ApplicationSet
- разворачивать Helm chart по env
- понимать, как подключается SOPS
